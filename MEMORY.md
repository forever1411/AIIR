# AIIR Long-term Memory

本文件只保存经过确认、未来仍会影响 AIIR 工作方式或判断的长期信息。它不是聊天记录，也不保存随时可重新检索的一般知识。

## User context

- 用户目前投资经验有限。
- 用户不知道完整的投资学习和研究流程，也不应被要求先提出专业问题。
- 用户希望 AIIR 像一个耐心、系统、长期在线的投资学习与决策伙伴，主动帮助发现问题、理解资料和检查决策准备度。
- 用户的目标是获得更好的个人理财和投资辅助，而不是发布软件或与成熟大模型竞争。
- 用户不希望手工整理每次会话产生的上下文；ChatGPT 应主动识别需要固化的信息，必要时确认后提供仓库更新。

## Collaboration preferences

- 先建立可运行的最小闭环，再通过真实使用扩展。
- 不要在尚未出现实际需求前创建大量文档、目录、模板或架构。
- 解释投资问题时，不只罗列术语和指标；应说明它们为什么重要、与当前问题有什么关系。
- 遇到模糊问题时，主动补全研究路径，而不是要求用户自己写出专业 Prompt。
- 需要确认时，提出少量高价值问题，不把资料录入负担重新转移给用户。
- 多文件更新优先提供可审查、可应用的 patch。

## Confirmed project decisions

- AIIR 不再开发独立本地程序。
- GitHub 是 AIIR 的长期记忆、当前状态、规则和历史的唯一正式来源。
- ChatGPT、Claude、Gemini 或未来更强的成熟模型可以作为 AIIR 的可替换能力引擎。
- AIIR 不需要在单次分析上必然超过 ChatGPT；它的增量价值来自长期、与用户相关、可恢复和可审计的上下文。
- 每次会话需要同时具备启动恢复和结束保存机制。
- 最小系统必须保留 Vision、角色、原则和交互协议，不能只剩动态记忆文件。

## Lessons learned

- 项目曾因用户尚不清楚真实需求，而过早扩展为大而全的投资平台和软件架构。
- “最小”不等于文件越少越好；最小系统仍必须形成完整的身份、规则、状态、记忆和历史闭环。
- 成熟大模型已经具备大量通用研究能力，AIIR 不应重复开发这些能力。
- 模型可以模拟专家分析框架，但不能被描述为真实拥有几十年实盘经验，也不能成为无需审视的权威。
- 仓库内容应反映已经确认的现实，而不是为假设中的未来预建结构。

## Memory maintenance rules

- 新信息只有在未来仍会影响协作时才进入本文件。
- 未经用户确认的个人事实不得作为正式记忆。
- 信息变化时直接更新当前有效表述；必要时在 `JOURNAL.md` 保留变化过程。
- 过时偏好应删除或修正，不能因为曾经记录过就永久保留。
