# 测试策略

## 1. Phase 0 测试

当前主要是人机流程测试，而不是代码测试。

测试材料包括：

- 三个真实用户场景；
- 一组固定模糊问题；
- 一组来源错误或冲突样例；
- 一次资产配置讨论；
- 一次决定记录；
- 一次复盘。

## 2. Prompt 回归

Prompt 修改后检查：

- 是否仍可从模糊问题开始；
- 是否主动提出关键问题；
- 是否减少而非增加信息过载；
- 是否正确区分事实和推断；
- 是否提出反方证据；
- 是否保持风险边界；
- 是否给出下一步；
- 是否泄露不必要个人信息。

## 3. 未来代码测试

### Unit

- 数据校验；
- 仓位和比例计算；
- 时间和版本比较；
- 上下文最小化；
- Schema 转换。

### Integration

- 私有数据导入导出；
- SQLite 迁移；
- 模型 Gateway；
- 来源元数据；
- 决策日志追加和复盘关联。

### Failure

- 模型超时；
- 模型返回非法 JSON；
- 来源不可用；
- 数据过期；
- 重复导入；
- 文件损坏；
- 缺失个人背景；
- 日志写入失败。

### Privacy

- 真实个人字段不进入 Git；
- 密钥不进入日志；
- 模型请求只包含允许字段；
- 测试 fixture 全部虚构；
- 删除和导出可验证。

## 4. 财务计算

所有计算测试使用固定输入和精确期望值。模型不得作为计算结果 Oracle。

## 5. 验收测试

每个功能必须使用最初触发开发的真实 friction 场景验收，并与未开发前的工作流比较。
