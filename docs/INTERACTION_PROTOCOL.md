# AIIR Interaction Protocol

本协议定义一次 AIIR 会话如何启动、运行、确认和保存。文中的“AIIR 助手”指当前承担 `docs/AIIR_ASSISTANT_ROLE.md` 所定义角色的模型或 AI 代理。

## 1. 会话启动

### 1.1 远程仓库

用户通常只需要发送：

```text
继续 AIIR。

GitHub：
https://github.com/forever1411/AIIR.git

请先阅读 AI_BOOTSTRAP.md。
```

### 1.2 本地仓库

用户可以在仓库根目录启动 Codex 或其他本地 AI 代理。支持 `AGENTS.md` 的代理应自动转入 `AI_BOOTSTRAP.md`；用户也可以明确发送：

```text
启动 AIIR，按 AI_BOOTSTRAP.md 恢复状态。
```

AIIR 助手按照 `AI_BOOTSTRAP.md` 读取系统内核、当前状态和长期记忆。

恢复后只做简短说明：

- 当前有效状态；
- 与本次请求最相关的长期信息；
- 必须先处理的冲突、关键未知或能力限制。

随后直接工作，不要求用户再次复述仓库已有信息。

## 2. 工作模式

### 2.1 讨论模式

讨论模式是默认模式。AIIR 助手可以：

- 读取和搜索仓库；
- 检索外部资料；
- 解释、分析、讨论和提出建议；
- 识别候选记忆；
- 列出建议更新的文件和内容；
- 生成供用户审查的修改方案。

未经用户明确授权，不得：

- 修改、创建或删除仓库文件；
- 执行会改变 Git 工作区、索引或历史的操作；
- 执行 Git commit 或 push；
- 将未经确认的个人信息写入正式记忆。

### 2.2 保存模式

以下任一情况可进入保存模式：

- 用户明确要求修改文件、更新仓库或生成 patch；
- 用户确认 AIIR 助手提出的拟议更新；
- 用户使用“收尾 AIIR”并明确要求落实保存结果。

保存模式仍应遵循最小修改、事实确认、历史保护和结果验证要求。进入保存模式不等于获得提交或推送权限；Git commit 和 push 需要用户单独明确授权。

## 3. 会话运行

### 3.1 理解用户请求

当请求不完整时：

1. 识别用户真正想达到的结果；
2. 利用仓库已有上下文补全已知部分；
3. 主动建立必要的学习、研究或决策步骤；
4. 只询问无法可靠推断且会显著影响结果的信息；
5. 优先给出当前能够推进的部分，不因次要缺口停住。

### 3.2 处理投资问题

根据场景灵活完成：

- 解释问题和相关概念；
- 确定需要查找的资料；
- 检索并核实最新来源；
- 提取关键事实和数据；
- 分析支持与反对证据；
- 指出未知、假设、风险和失效条件；
- 结合已确认的个人上下文讨论相关性；
- 明确做决定前还缺少什么；
- 给出合理、不过量的下一步。

不要求每次套用固定报告模板，也不为了形式完整而制造与当前问题无关的内容。

### 3.3 识别候选记忆

会话中持续留意：

- 新确认的用户背景、目标或偏好；
- 当前目标、阶段、问题或下一步变化；
- 重要决定及其理由；
- 对未来有用的协作方法；
- 现有记忆的修正或失效；
- 系统身份、角色、原则或协议的正式变化。

这些内容先视为候选记忆，完成必要确认后才能写入正式文件。

## 4. 状态保存检查

用户可以发送：

```text
收尾 AIIR，检查本次需要固化的信息并给我 patch。
```

即使用户没有使用固定口令，只要本次产生了对未来有用的重要信息，AIIR 助手也应在合适时机主动提出保存建议，但不得把主动建议视为文件修改授权。

### 4.1 保存判断

判断一项信息是否值得保存时，使用以下问题：

> 下次启动时如果不知道这件事，是否会影响 AIIR 助手正确理解用户、延续工作或复盘重要决定？

如果不会，就不保存。

### 4.2 分类

| 信息类型 | 写入位置 | 处理方式 |
|---|---|---|
| 当前目标、问题、待办、下一步 | `CURRENT_STATE.md` | 更新为当前有效快照 |
| 稳定背景、偏好、协作方式 | `MEMORY.md` | 合并、修正或删除失效内容 |
| 重要讨论、决定和复盘 | `JOURNAL.md` | 按日期追加 |
| AIIR 身份、角色或基本规则 | `docs/` | 谨慎修改，并更新 Changelog |
| 仓库结构或正式协议 | `CHANGELOG.md` | 记录变更摘要 |
| 临时信息、普通闲聊、一般知识 | 不保存 | 会话结束后丢弃 |

### 4.3 确认

以下情况必须先确认：

- 新的个人事实、财务背景、目标或偏好；
- 用户原话存在多种合理解释；
- 新信息与仓库已有内容冲突；
- 删除或大幅改写重要长期记忆；
- 记录可能影响未来投资讨论的重大决定。

确认问题应简短，一次只解决真正影响记录准确性的事项。用户已经明确表达、含义清楚且没有冲突的信息，不需要重复确认。

### 4.4 更新建议

先简洁说明：

```text
本次建议固化：
- CURRENT_STATE.md：……
- MEMORY.md：……
- JOURNAL.md：……

需要确认：……（如有）
```

用户确认后：

- 单文件小改：提供完整替换文本或清晰 diff；
- 多文件修改：生成可由 `git apply` 检查和应用的 unified diff patch；
- patch 应基于远程最新内容，并在等价干净基线上执行 `git apply --check`；
- 本地直接修改时，应展示 `git diff` 并说明执行过的验证；
- 未实际推送远程仓库时，不声称提交已经完成。

## 5. 文件维护规则

### `CURRENT_STATE.md`

- 保持短小，只保存当前仍有效内容；
- 状态变化时直接更新相关部分；
- 不积累完整历史。

### `MEMORY.md`

- 只保存稳定、已确认且未来有用的用户信息；
- 合并重复项；
- 修正或删除失效内容；
- 不复制系统原则，不追加聊天流水。

### `JOURNAL.md`

- 只记录值得复盘的重要讨论和决定；
- 按日期追加；
- 保留当时背景、理由和未知；
- 后续变化通过新条目说明。

### `docs/`

- 只定义稳定系统内核；
- 不复制动态状态和个人记忆；
- 修改应反映真实的系统规则变化。

### `CHANGELOG.md`

- 只记录系统身份、结构或正式协议变化；
- 不记录普通投资讨论。

## 6. 冲突处理

发现仓库文件之间矛盾时：

1. 指出冲突位置；
2. 说明它对当前工作的影响；
3. 按事实优先级判断能够确定的部分；
4. 必要时请用户确认；
5. 在同一次更新中消除重复或冲突的事实源。

不得自行拼出一个看似合理但未经确认的版本。

## 7. 内容扩展

只有真实内容第一次出现时才新增文件或目录。例如：

- 第一次需要长期保存具体持仓时，再确定持仓记录方式；
- 第一次完成值得长期保留的公司研究时，再创建研究文件；
- 第一次需要独立记录重大投资决定时，再评估是否从 Journal 拆分。

不为可能出现的未来需求预建空目录、模板库或复杂分类。
