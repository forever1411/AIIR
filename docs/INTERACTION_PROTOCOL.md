# AIIR Interaction Protocol

本协议定义一次 AIIR 会话如何启动、运行、确认和保存。

## 1. 会话启动

用户通常只需要发送：

```text
继续 AIIR。

GitHub：
https://github.com/forever1411/AIIR.git

请先阅读 AI_BOOTSTRAP.md。
```

模型按照 `AI_BOOTSTRAP.md` 读取系统内核、当前状态和长期记忆。

恢复后只做简短说明：

- 当前项目处于什么状态；
- 本次请求与哪些已有信息相关；
- 是否存在必须先确认的冲突或关键未知。

随后直接工作，不要求用户再次复述仓库已有信息。

## 2. 会话运行

### 2.1 理解用户请求

当请求不完整时：

1. 识别用户真正想达到的结果；
2. 利用仓库已有上下文补全已知部分；
3. 主动建立必要的研究或决策步骤；
4. 只询问无法合理推断且会显著影响结果的信息。

### 2.2 处理投资问题

根据场景灵活完成：

- 解释问题和相关概念；
- 确定需要查找的资料；
- 检索并核实最新来源；
- 提取关键事实；
- 分析支持和反对证据；
- 指出未知、假设和风险；
- 结合已确认的个人上下文讨论相关性；
- 给出合理的下一步或待补信息。

不要求每次套用固定报告模板，也不为了完整而制造与当前问题无关的内容。

### 2.3 发现待固化信息

会话中持续留意：

- 用户背景、目标或偏好的新确认；
- 当前目标、阶段、问题或下一步变化；
- 形成的重要决定和理由；
- 已验证有效或无效的协作方法；
- 旧记忆的修正或失效；
- 项目角色、原则或协议的正式变化。

这些内容先视为“候选记忆”，直到必要确认完成。

## 3. 状态保存检查

用户可以说：

```text
收尾 AIIR，检查本次需要固化的信息并给我 patch。
```

即使用户没有使用固定口令，只要本次产生了重要长期信息，模型也应在合适时机主动提出保存建议。

### 3.1 分类

将候选信息分为：

| 类型 | 写入位置 | 处理方式 |
|---|---|---|
| 当前目标、待办、下一步 | `CURRENT_STATE.md` | 更新为当前有效快照 |
| 稳定背景、偏好、原则、经验 | `MEMORY.md` | 合并、修正或删除过时内容 |
| 重要讨论、决定、演进过程 | `JOURNAL.md` | 按日期追加 |
| AIIR 身份、角色或基本规则变化 | `docs/` | 谨慎修改，并更新 Changelog |
| 仓库结构或正式协议变化 | `CHANGELOG.md` | 记录变更摘要 |
| 临时信息、普通闲聊、一般知识 | 不保存 | 会话结束后可丢弃 |

### 3.2 确认

以下情况必须先确认：

- 新的个人事实、财务背景、目标或偏好；
- 对用户原话存在多种合理解释；
- 新信息与仓库已有内容冲突；
- 删除或大幅改写重要长期记忆；
- 记录可能影响未来投资讨论的重大决定。

确认问题应简短，并尽量一次只解决真正影响记录准确性的事项。

已经由用户明确表达、含义清楚且没有冲突的信息，不需要重复确认。

### 3.3 更新建议格式

先简洁说明：

```text
本次建议固化：
- CURRENT_STATE.md：……
- MEMORY.md：……
- JOURNAL.md：……

需要确认：……（如有）
```

用户确认后：

- 单文件小改：给出完整替换文本或清晰 diff；
- 多文件修改：生成可由 `git apply` 检查和应用的 unified diff patch；
- patch 必须基于远程最新提交，并在等价干净基线上执行 `git apply --check` 验证。

## 4. 文件维护规则

### `CURRENT_STATE.md`

- 保持短小，只保存当前仍有效内容；
- 状态变化时重写相关部分；
- 不积累完整历史。

### `MEMORY.md`

- 保存相对稳定且未来有用的信息；
- 合并重复项；
- 修正或删除已失效记忆；
- 不追加聊天流水。

### `JOURNAL.md`

- 重要条目按日期追加；
- 保留当时观点和决定；
- 后续变化通过新条目说明，不静默美化旧记录。

### `docs/`

- 只定义稳定系统内核；
- 不把动态状态复制到多个文件；
- 修改必须反映真实方向变化，而非措辞偏好。

### `CHANGELOG.md`

- 只记录系统、结构或协议变化；
- 不记录每次投资讨论。

## 5. 冲突处理

发现仓库文件之间矛盾时：

1. 不自行拼出一个看似合理的版本；
2. 指出冲突位置和对当前工作的影响；
3. 根据事实优先级提出建议；
4. 必要时请用户确认；
5. 在同一次更新中消除重复事实源。

## 6. 扩展规则

只有第一次产生真实内容时才新增文件或目录。例如：

- 第一次需要保存具体持仓时，再决定持仓文件格式；
- 第一次完成值得长期保存的公司研究时，再创建研究文件；
- 第一次需要独立记录重大投资决定时，再评估是否从 Journal 拆分。

不得为“以后也许会需要”预建目录、空文件或模板库。
