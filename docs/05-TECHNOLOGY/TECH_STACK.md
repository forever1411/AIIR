# 技术栈策略

## 1. 当前技术栈

Phase 0 不需要应用技术栈。

当前使用：

- ChatGPT Project；
- Markdown；
- CSV；
- Git 管理公开项目文档；
- Git 忽略的本地私有目录管理真实个人数据；
- Codex、Claude Code 或其他本地代理用于已批准的仓库工作。

## 2. 首个软件实验的默认选择

只有进入 Phase 1 后才采用：

| 层 | 默认选择 | 原因 |
|---|---|---|
| 语言 | Python 3.12+ | 数据和模型生态成熟，用户本地环境适合 |
| 接口 | CLI | 最低成本、便于验证、无前端负担 |
| 数据 | SQLite + Markdown/CSV 导入导出 | 本地、简单、可备份、可查询 |
| 配置 | 环境变量 + TOML/YAML | 清晰且可版本化非敏感配置 |
| 模型 | 一个 Provider，通过最小 Adapter | 避免锁定但不提前多模型化 |
| 校验 | Pydantic 或标准 dataclass + 显式校验 | 结构化输入输出 |
| 测试 | pytest | 简单成熟 |
| 质量 | ruff，必要时 mypy | 快速反馈 |
| 打包 | `pyproject.toml` | 标准 Python 项目 |

## 3. 非默认选择

以下工具只有出现明确需求和 ADR 后才能引入：

- FastAPI；
- PostgreSQL；
- Redis；
- Celery；
- Docker；
- 向量数据库；
- LangChain/LangGraph；
- Agent 框架；
- 浏览器自动化；
- 云部署；
- 前端框架。

它们不是禁止使用，而是不能因为“专业项目通常有”就提前引入。

## 4. 模型选择原则

- 根据当前任务质量、上下文、检索、结构化输出和成本选择；
- 不在核心数据中保存供应商专属对象；
- V0 软件只支持一个模型；
- 模型升级应通过评估样例，而不是仅凭发布时间；
- 可直接使用 ChatGPT 完成的交互，不强制转成 API 调用。

## 5. 数据源技术原则

- 官方公开来源优先；
- 遵守访问频率、授权和再分发限制；
- 先支持手工提供 URL，再考虑自动发现；
- 先保存元数据和原始文件，再调用模型；
- 不建设全市场爬虫作为第一步。

## 6. 技术选型门槛

每项新技术必须回答：

1. 它解决了哪条重复 friction？
2. 不使用它的最低成本方案是什么？
3. 它增加了哪些维护和隐私风险？
4. 两周内如何验证价值？
5. 如何删除或替换它？
